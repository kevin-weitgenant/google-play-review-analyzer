# Plan 4.2 (Phase 2) — AI Analysis: 2 Prompts per Review via Groq

## Goal

Layer AI analysis on top of the existing scraper. Each review gets **2 separate Groq API calls** — one for **sentiment** (positive / neutral / negative), one for **priority** (high / medium / low).

The prompts come from the **frontend** in the request body. The AI analysis is a **service only** — no separate route. A new endpoint in the existing **reviews router** orchestrates the flow: scrape → analyze → return.

---

## Architecture

```
Frontend
  │
  ├─ POST /api/reviews/fetch
  │     └── Scraper only (exists already, no changes)
  │
  └─ POST /api/reviews/fetch-and-analyze  ← NEW
        │
        ├── 1. Scraper service → fetches reviews
        ├── 2. Groq service → 2 calls per review
        │     ├── analyze_sentiment(content, instructions) → str
        │     └── analyze_priority(content, instructions) → str
        └── 3. Returns merged response
```

- **Scraper service** — `app/services/scraper.py` (already exists)
- **Groq service** — `app/services/groq_service.py` (new, service only, no route)
- **Reviews router** — `app/routers/reviews.py` (new endpoint added here)
- **No `analysis.py` router** — AI analysis is not a route, it's a service called by the reviews router

---

## Key Design Decisions

### Model: `llama-3.1-8b-instant`

| | Value |
|---|---|
| Model ID | `llama-3.1-8b-instant` |
| Speed | 560 tokens/sec |
| Cost | $0.05 input / $0.08 output per 1M tokens |
| Context window | 131,072 tokens |

Cheapest option. Sentiment (3-class) and priority (3-class) are trivial classification tasks. Cost for 25 reviews × 2 calls = ~$0.00008 per request.

### 2 separate prompts per review

Each review triggers **2 independent Groq API calls**:

| Call | Returns | Structured JSON |
|---|---|---|
| Sentiment prompt | `{"sentiment": "positive" \| "neutral" \| "negative"}` | Yes |
| Priority prompt | `{"priority": "high" \| "medium" \| "low"}` | Yes |

Each prompt is focused and simple — the LLM is less likely to get confused. They're independently swappable (different wording, can use different models later).

### Prompts: backend owns the structure, frontend adds nuance

The prompt is **always** built by the backend. The frontend only provides a **custom instruction substring** — extra detail about what to prioritize or how to interpret things.

The full prompt sent to Groq is assembled like this:

```
[Backend fixed header — task description + JSON format rules]

[Frontend custom instructions — optional extra context]

[Backend fixed footer — the review text]
```

The frontend **cannot** change:
- What the task is (sentiment or priority classification)
- That the output must be JSON
- The expected JSON schema

The frontend **can** add:
- Extra context (e.g. "pay special attention to crash reports and login issues")
- Nuanced rules (e.g. "reviews mentioning pricing alone should be neutral, not negative")
- Domain-specific guidance

If the frontend sends nothing, the prompt still works — the custom section is just empty.

### Default vs. Custom Instructions — What the LLM Receives

**If the user sends NO custom instructions** (both fields omitted or empty), here's exactly what goes to Groq:

*Sentiment call — full prompt:*
```
You are a sentiment classifier for app reviews. Classify the review below as exactly one of: "positive", "neutral", or "negative".

Respond with a JSON object: {"sentiment": "positive"|"neutral"|"negative"}

Review:
I like the game and it's cookies...
```

*Priority call — full prompt:*
```
You are a priority classifier for app reviews. Classify the review below as exactly one of: "high", "medium", or "low".
"high" = urgent issue that needs immediate attention (bug, crash, data loss, security vulnerability, app unusable).
"medium" = notable problem but not critical (performance issues, minor bugs, feature requests for existing features).
"low" = general feedback, praise, complaints without actionable detail, or non-urgent suggestions.

Respond with a JSON object: {"priority": "high"|"medium"|"low"}

Review:
I like the game and it's cookies...
```

That's it. The default prompts classify sentiment into 3 buckets and priority into 3 buckets. No extra rules.

**If the user DOES add custom instructions**, the only difference is an extra paragraph sandwiched in the middle:

```
[...fixed header...]

[USER'S CUSTOM INSTRUCTIONS INSERTED HERE]

[...fixed footer with the review text...]
```

For example, if the user sends:
```json
{
  "sentiment_instructions": "Reviews about pricing alone should be neutral, not negative.",
  "priority_instructions": "Crash reports and login issues are always high priority."
}
```

Then the LLM sees the user's text as an extra paragraph in the middle. The header and footer never change.

### Only the comment text goes to the LLM

```
✅ Sent to Groq:  content: "the servers are terrible, with awful lag..."
❌ Never sent:    reviewId, userName, userImage, score, thumbsUpCount,
                  reviewCreatedVersion, at, replyContent, repliedAt, appVersion
```

### Structured JSON output via Groq `json_schema` (strict mode)

Groq guarantees the response matches the schema exactly:

```json
// Sentiment call returns:
{"sentiment": "positive"}

// Priority call returns:
{"priority": "high"}
```

### Tenacity — exponential backoff

Both Groq functions are wrapped with `@retry`:

```python
@retry(
    wait=wait_exponential(multiplier=1, min=1, max=10),
    stop=stop_after_attempt(3),
    retry=retry_if_exception_type((httpx.HTTPStatusError, httpx.ConnectError)),
)
```

- Backoff: 1s → 2s → 4s (capped at 10s)
- 3 attempts then give up
- `return_exceptions=True` in `asyncio.gather` → one failure degrades gracefully (defaults to `neutral` / `low`)

### 2. `app/services/groq_service.py` — The AI service (2 functions)

Replace the placeholder comment with the full implementation. This is a **service only** — no router, no endpoint.

```python
import json

import httpx
from tenacity import (
    retry,
    wait_exponential,
    stop_after_attempt,
    retry_if_exception_type,
)

from app.core.config import settings
from app.schemas.api import (
    SENTIMENT_PROMPT_HEADER,
    SENTIMENT_PROMPT_FOOTER,
    PRIORITY_PROMPT_HEADER,
    PRIORITY_PROMPT_FOOTER,
)

GROQ_CHAT_URL = "https://api.groq.com/openai/v1/chat/completions"

SENTIMENT_JSON_SCHEMA = {
    "type": "json_schema",
    "json_schema": {
        "name": "sentiment",
        "strict": True,
        "schema": {
            "type": "object",
            "properties": {
                "sentiment": {
                    "type": "string",
                    "enum": ["positive", "neutral", "negative"],
                }
            },
            "required": ["sentiment"],
            "additionalProperties": False,
        },
    },
}

PRIORITY_JSON_SCHEMA = {
    "type": "json_schema",
    "json_schema": {
        "name": "priority",
        "strict": True,
        "schema": {
            "type": "object",
            "properties": {
                "priority": {
                    "type": "string",
                    "enum": ["high", "medium", "low"],
                }
            },
            "required": ["priority"],
            "additionalProperties": False,
        },
    },
}


@retry(
    wait=wait_exponential(multiplier=1, min=1, max=10),
    stop=stop_after_attempt(3),
    retry=retry_if_exception_type((httpx.HTTPStatusError, httpx.ConnectError)),
)
async def analyze_sentiment(
    client: httpx.AsyncClient,
    content: str,
    instructions: str = "",
) -> str:
    """Classify review sentiment. Returns 'positive', 'neutral', or 'negative'.

    Args:
        client: Shared httpx async client for connection pooling.
        content: The review comment text — nothing else.
        instructions: Optional extra instructions from the frontend.
    """
    # Assemble: header + optional instructions + footer (with review text)
    parts = [SENTIMENT_PROMPT_HEADER]
    if instructions:
        parts.append(instructions)
    parts.append(SENTIMENT_PROMPT_FOOTER.format(content=content))
    user_message = "\n\n".join(parts)

    response = await client.post(
        GROQ_CHAT_URL,
        headers={"Authorization": f"Bearer {settings.groq_api_key}"},
        json={
            "model": settings.groq_model,
            "messages": [{"role": "user", "content": user_message}],
            "temperature": 0,
            "max_tokens": 50,
            "response_format": SENTIMENT_JSON_SCHEMA,
        },
    )
    response.raise_for_status()

    raw = response.json()["choices"][0]["message"]["content"]
    result = json.loads(raw)
    return result["sentiment"]


@retry(
    wait=wait_exponential(multiplier=1, min=1, max=10),
    stop=stop_after_attempt(3),
    retry=retry_if_exception_type((httpx.HTTPStatusError, httpx.ConnectError)),
)
async def analyze_priority(
    client: httpx.AsyncClient,
    content: str,
    instructions: str = "",
) -> str:
    """Classify review priority. Returns 'high', 'medium', or 'low'.

    Args:
        client: Shared httpx async client for connection pooling.
        content: The review comment text — nothing else.
        instructions: Optional extra instructions from the frontend.
    """
    # Assemble: header + optional instructions + footer (with review text)
    parts = [PRIORITY_PROMPT_HEADER]
    if instructions:
        parts.append(instructions)
    parts.append(PRIORITY_PROMPT_FOOTER.format(content=content))
    user_message = "\n\n".join(parts)

    response = await client.post(
        GROQ_CHAT_URL,
        headers={"Authorization": f"Bearer {settings.groq_api_key}"},
        json={
            "model": settings.groq_model,
            "messages": [{"role": "user", "content": user_message}],
            "temperature": 0,
            "max_tokens": 50,
            "response_format": PRIORITY_JSON_SCHEMA,
        },
    )
    response.raise_for_status()

    raw = response.json()["choices"][0]["message"]["content"]
    result = json.loads(raw)
    return result["priority"]
```

Key details:
- **2 separate functions** — each is a focused, single-responsibility call
- **Both decorated with `@retry`** — tenacity exponential backoff on transient errors
- **`instructions: str`** — optional substring from the frontend, inserted between header and footer
- **`content: str`** — only the review comment text
- **Prompt assembled from 3 parts** — `[HEADER] + [instructions] + [FOOTER.format(content)]`
- **`json_schema` strict mode** — Groq guarantees valid JSON matching the schema
- **`temperature: 0`** — deterministic classification
- **Accepts `httpx.AsyncClient`** — caller manages the client for connection pooling

New endpoint:

```python
@router.post("/fetch-and-analyze", response_model=FetchAndAnalyzeResponse)
async def fetch_and_analyze(request: FetchAndAnalyzeRequest):
    # 1. Scrape reviews
    try:
        parsed = extract_app_id(request.url)
    except ValueError as e:
        raise HTTPException(status_code=422, detail=str(e))

    app_id = parsed["app_id"]
    lang = parsed["lang"]
    country = parsed["country"]

    try:
        app_name = get_app_name(app_id, lang=lang, country=country)
    except Exception as e:
        raise HTTPException(status_code=502, detail=f"Failed to fetch app details: {e}")

    try:
        raw_reviews = fetch_reviews_paginated(
            app_id=app_id,
            lang=lang,
            country=country,
            sort=request.sort,
            filter_score=request.filter_score,
            count=settings.max_reviews_to_fetch,
        )
    except ValueError as e:
        raise HTTPException(status_code=422, detail=str(e))
    except Exception as e:
        raise HTTPException(status_code=502, detail=f"Failed to fetch reviews: {e}")

    # 2. Convert to API response models
    reviews = [_to_response(r) for r in raw_reviews]

    # 3. Optional date filter
    if request.since:
        reviews = [r for r in reviews if r.at >= request.since]

    # 4. AI analysis — 2 calls per review, all concurrent
    #    Layout: [sent_1, prio_1, sent_2, prio_2, ..., sent_N, prio_N]
    async with httpx.AsyncClient() as client:
        tasks = []
        for review in reviews:
            tasks.append(
                analyze_sentiment(
                    client,
                    review.content,
                    instructions=request.sentiment_instructions,
                )
            )
            tasks.append(
                analyze_priority(
                    client,
                    review.content,
                    instructions=request.priority_instructions,
                )
            )

        results = await asyncio.gather(*tasks, return_exceptions=True)

    # 5. Merge AI results into review objects
    analyzed = []
    for i, review in enumerate(reviews):
        sent_result = results[i * 2]       # sentiment call
        prio_result = results[i * 2 + 1]   # priority call

        # Graceful degradation on errors
        sentiment = sent_result if isinstance(sent_result, str) else "neutral"
        priority = prio_result if isinstance(prio_result, str) else "low"

        analyzed.append(
            AnalyzedReviewResponse(
                **review.model_dump(),
                sentiment=sentiment,
                priority=priority,
            )
        )

    return FetchAndAnalyzeResponse(
        app_id=app_id,
        app_name=app_name,
        total_analyzed=len(analyzed),
        reviews=analyzed,
    )
```

Key details:
- **2 calls per review** — `analyze_sentiment()` + `analyze_priority()`, both fire in parallel
- **50 concurrent async calls** (25 reviews × 2) via `asyncio.gather`
- **`return_exceptions=True`** — one Groq failure → defaults to `neutral` / `low`, doesn't kill the batch
- **Interleaved results** — `[sent_1, prio_1, sent_2, prio_2, ...]`, indexed as `results[i*2]` and `results[i*2+1]`
- **Single `httpx.AsyncClient`** — connection pooling across all 50 calls
- **Prompts from request** — `request.sentiment_instructions` and `request.priority_instructions` are optional substrings
- Reuses existing `_to_response()`, `extract_app_id()`, `fetch_reviews_paginated()`, `get_app_name()`

## End-to-End Flow — What Happens from Request to Response

This section explains the **full lifecycle** of a single `POST /api/reviews/fetch-and-analyze` request. No SSE, no streaming — simple request/response.

### Step 1: User sends a request

The frontend sends a single POST request. Two scenarios:

**Scenario A — No custom instructions (defaults):**
```json
POST /api/reviews/fetch-and-analyze
{
    "url": "https://play.google.com/store/apps/details?id=com.devsisters.cos",
    "sort": "newest"
}
```
Both `sentiment_instructions` and `priority_instructions` are empty strings (the default). The backend uses only the fixed header + footer for each prompt. See "Default vs. Custom Instructions" above for the exact prompts.

**Scenario B — With custom instructions:**
```json
POST /api/reviews/fetch-and-analyze
{
    "url": "https://play.google.com/store/apps/details?id=com.devsisters.cos",
    "sort": "newest",
    "sentiment_instructions": "Reviews about pricing alone should be neutral, not negative.",
    "priority_instructions": "Crash reports and login issues are always high priority."
}
```
The user's instructions get sandwiched between the fixed header and footer. See "Default vs. Custom Instructions" above.

### Step 2: Backend scrapes reviews (sequential, before any AI calls)

The endpoint calls the existing scraper service. This is synchronous within the endpoint — no AI calls happen until all reviews are fetched. If the scraper fails, the endpoint returns an error immediately. No Groq calls are wasted.

### Step 3: Backend fires all Groq calls in parallel

For N reviews, the backend creates **2N async tasks**:
- `analyze_sentiment(review_1.content)` —→ Groq API call #1
- `analyze_priority(review_1.content)`  —→ Groq API call #2
- `analyze_sentiment(review_2.content)` —→ Groq API call #3
- `analyze_priority(review_2.content)`  —→ Groq API call #4
- ... and so on up to review N

All 2N calls are fired **at the same time** via `asyncio.gather(*tasks, return_exceptions=True)`. The backend waits for **all of them** to complete (or fail) before proceeding. There is no streaming — the frontend is waiting this entire time.

Each individual Groq call sends only:
```
{
    "model": "llama-3.1-8b-instant",
    "messages": [{"role": "user", "content": "<assembled prompt>"}],
    "temperature": 0,
    "max_tokens": 50,
    "response_format": { <json_schema for sentiment or priority> }
}
```

The `<assembled prompt>` is the header + optional instructions + footer (with the review text). **No metadata** (review_id, score, user name, etc.) is ever sent to Groq — only the plain review text.

### Step 4: What happens if a Groq API call fails

Each individual call has tenacity retry: up to 3 attempts with exponential backoff (1s → 2s → 4s). If it still fails after 3 attempts:

- `return_exceptions=True` means the exception is **captured as a result** instead of crashing the whole batch
- For that specific review, the backend checks `isinstance(result, str)`:
  - If the result is a valid string → use it (it's "positive"/"neutral"/"negative" or "high"/"medium"/"low")
  - If the result is an exception → **fall back to a safe default**
    - Sentiment default: `"neutral"`
    - Priority default: `"low"`

**Key point:** One review's failure does NOT affect any other review. All other reviews get their real results. Only the failed review gets the default. Even if ALL reviews fail, the response still returns — just with all defaults.

### Step 5: Backend merges results and returns one JSON response

Once all 2N calls are done (or timed out), the backend merges each review with its AI results and returns **a single JSON response**:

```json
{
    "app_id": "com.devsisters.cos",
    "app_name": "CookieRun: OvenSmash",
    "total_analyzed": 25,
    "reviews": [
        {
            "review_id": "31d6eaaa-...",
            "user_name": "Christian James",
            "user_image": "https://...",
            "content": "I like the game and it's cookies...",
            "score": 5,
            "thumbs_up_count": 0,
            "review_created_version": null,
            "at": "2026-04-25T16:47:33",
            "reply_content": null,
            "replied_at": null,
            "app_version": null,
            "sentiment": "positive",
            "priority": "low"
        },
        {
            "review_id": "8daec61b-...",
            "user_name": "Edisiinn",
            "user_image": "https://...",
            "content": "...the servers are terrible, with awful lag...",
            "score": 1,
```

**What the user receives:** One complete response with all reviews, each annotated with `sentiment` and `priority`. The frontend gets everything at once — no partial updates, no SSE, no WebSocket. Simple HTTP request → wait → full JSON response.

### Summary: the lifecycle in one picture

```
Frontend sends POST /fetch-and-analyze
         │
         ▼
Backend scrapes reviews (sequential)
         │
         ▼
Backend fires 2N Groq calls (all at once, parallel)
  ┌──────┼──────┐
  │      │      │
  ▼      ▼      ▼
sent_1  prio_1  ...  (each with retry x3)
  │      │      │
  └──────┼──────┘
         │
         ▼  (wait for ALL to finish)
         │
Backend merges results (fallback to defaults for failures)
         │
         ▼
Backend returns ONE JSON response to frontend
```

No SSE. No streaming. Simple and reliable.

## Error handling summary

| Scenario | Behavior |
|---|---|
| Groq returns 429 (rate limit) | Tenacity retries with exponential backoff (1s → 2s → 4s), up to 3 attempts |
| Groq returns 5xx | Same retry behavior |
| Groq returns invalid JSON | `json.loads` raises, caught by `return_exceptions`, defaults to neutral/low |
| Single review fails after 3 retries | That review gets `sentiment: "neutral"`, `priority: "low"` |
| All reviews fail | Response still has all reviews, just with default values |
| Scraper fails | Returns HTTP 502 before any Groq calls are made |
| Invalid URL | Returns HTTP 422 before any Groq calls are made |

## Out of scope (future)

- **Caching** — same review analyzed twice wastes Groq calls. Could cache by `review_id`.
- **Streaming (SSE)** — stream results to frontend as they arrive instead of waiting for all 25.
- **Batching** — send multiple reviews per request instead of one-by-one.
- **Database persistence** — store analysis results in Supabase for historical views.
- **User-adjustable model** — let frontend choose model via request param.
